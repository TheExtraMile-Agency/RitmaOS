## Core Framework

Module 5.3 turns 60/30/10 into a repeatable diagnostic and training-funnel system.

The reusable flow is:

```text
real workflow -> layer audit -> bottleneck diagnosis -> leverage estimate -> flywheel map -> pattern ownership
```

The framework becomes owned through practice on real work, not by memorizing the ratio.

## 60/30/10 Workflow Audit

The audit maps a repeated process into:

| Layer | Diagnostic Focus |
|---|---|
| 60% Infrastructure | Systems, databases, file storage, routing, approvals, and durable structures that should remain or be built. |
| 30% Orchestration | Connections, formatting logic, routing rules, decision logic, and context-specific workflow structure. |
| 10% AI | Summarization, extraction, generation, comparison, and other model-appropriate tasks. |

The audit asks what already exists, what is missing, and what manual work belongs in a different layer.

## Bottleneck Layer Diagnosis

The lesson frames bottlenecks as layer problems:

```text
A) missing infrastructure
B) missing orchestration
C) missing AI capability
```

This prevents premature tool selection. The output should be a structural change, not a generic tool recommendation.

## Training Engagement as Data Collection

Training exercises produce structured intake data because users apply the framework to their own work.

Reusable pattern:

```text
real user exercise -> bottleneck description -> layer diagnosis -> build opportunity
```

The training funnel surfaces where AI fits, where code fits, and where humans stay.

## Leverage Map

The leverage map estimates time savings task by task.

Inputs:

- recurring weekly tasks,
- rough time estimates,
- infrastructure percentage,
- orchestration percentage,
- AI-appropriate percentage.

Outputs:

- realistic best-case savings,
- conservative savings,
- tasks that do not compress,
- tasks that compress dramatically when layers are separated.

The goal is honest leverage, not inflated claims.

## Professional Flywheel

The flywheel pattern is:

```text
Training -> Build -> Case Study -> Content -> Next Client -> Training
```

Generalized:

```text
delivery -> byproduct data/insight -> improved next delivery -> proof/story -> next buyer or stakeholder
```

The useful diagnostic is where the loop leaks: unused data, unpublished proof, unconverted case studies, or outputs that do not feed future work.

## Tools vs Patterns

The lesson separates tool skills from pattern skills:

```text
tool skills decay when interfaces change
pattern skills survive tool changes
```

The productizable move is to identify the pattern underneath each tool-specific behavior.

Examples of pattern categories:

- task decomposition,
- context structuring,
- workflow routing,
- reusable prompts as process artifacts,
- separating infrastructure, orchestration, and AI.

## Framework Ownership

The framework becomes durable when users repeatedly diagnose new bottlenecks through the three layers until the categories become automatic.

Ownership pattern:

```text
prompted diagnosis -> repeated practice -> automatic layer recognition -> independent judgment
```

## RitmaOS Extraction

Module 5.3 suggests RitmaOS should capture:

- repeated bottlenecks,
- workflow patterns,
- layer diagnoses,
- leverage estimates,
- proof artifacts,
- next-build opportunities,
- training exercise outputs.

The file system should act as a structured intake and memory layer for future system builds, not only a content archive.
