## Core Framework

Module 5.2 applies the 60/30/10 ratio to business structure rather than a codebase.

The reusable frame is:

```text
business infrastructure -> orchestration loop -> AI capability
```

The lesson treats AI as the enabling technology, not the business model.

## Partnership Layer Map

The business-layer translation is:

| Layer | Business Meaning |
|---|---|
| 60% Infrastructure | Contracts, payment structures, invoicing systems, deliverable templates, content libraries, operational foundations. |
| 30% Orchestration | How one stream of work feeds the next: training, pain points, specs, builds, case studies, content, and future clients. |
| 10% AI | The technology capability that delivers or enhances the work, but does not carry the whole relationship. |

The removal test is central:

```text
If AI were swapped for another technology, would the business model still work?
```

If yes, the user has architecture. If no, the user has a feature demo.

## Multi-Stream Partnership Model

The NLP Logix pattern converts one relationship into several connected revenue streams.

Example stream types:

- retainer,
- day rates,
- revenue share,
- course delivery,
- consulting,
- content production.

The reusable lesson is that these are not separate products. They are streams produced by the structure of the relationship.

## Revenue Stream Mapper

The revenue stream exercise follows this pattern:

```text
current paid work -> uncharged capabilities -> reachable buyers -> possible streams -> infrastructure needed -> stream connections
```

The goal is not to maximize the number of streams. The goal is to identify the streams that connect and compound.

## In-House Comparison as Value Anchor

The in-house comparison positions a service against the cost and delay of internal hiring.

Reusable comparison:

| Side | Inputs |
|---|---|
| In-house | Required roles, salaries, ramp time, management overhead, tools, iteration, total annual cost. |
| External architecture | Retainer or fee, first deliverable timing, existing templates, proven process, experience, reduced ramp time. |

The value anchor is the honest cost reduction and speed advantage, not a vague AI capability claim.

## Orchestration Loop / Flywheel

The core orchestration loop from the lesson is:

```text
training -> pain points -> build specs -> builds -> case studies -> content -> next client
```

The general pattern is:

```text
main work -> proof -> data/byproducts -> next engagement -> reusable asset -> next buyer
```

Leaks happen when proof, data, templates, case studies, or relationships are created but not routed anywhere.

## Sell Architecture, Not AI

The client conversation pattern starts with the client's infrastructure and orchestration problems.

Reusable process:

1. Ask about the client's business before pitching.
2. Surface bottlenecks in delivery, operations, client demand, or internal capacity.
3. Identify where structure is missing.
4. Let the offering emerge from the diagnosis.
5. Introduce AI only after the 60% and 30% are visible.

The lesson reframes selling from a capability demo into a structural conversation.

## RitmaOS Extraction

Module 5.2 suggests RitmaOS should expose how work compounds across:

- clients,
- relationships,
- proof assets,
- templates,
- training materials,
- case studies,
- content,
- future offers.

The filesystem should not only organize current work. It should make the orchestration loop visible so byproducts can be routed into future value instead of disappearing after delivery.
