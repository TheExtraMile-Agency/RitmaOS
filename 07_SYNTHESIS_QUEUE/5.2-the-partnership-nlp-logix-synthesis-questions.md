## Questions

- Should RitmaOS include business-structure workflows in the core product, or stay focused on workspace and system architecture?
- Should Partnership Layer Map become a named RitmaOS workflow?
- How should RitmaOS model business infrastructure separately from technical infrastructure?
- Should Revenue Stream Mapper be part of the same workflow as Partnership Layer Map or a separate product surface?
- How should RitmaOS prevent users from forcing revenue streams that only work in theory?
- Should in-house cost comparisons become reusable client-facing artifacts?
- How should RitmaOS distinguish honest value anchoring from misleading sales comparisons?
- Should orchestration loops be stored as living workflow maps that update after every project?
- How should proof assets, case studies, templates, content, and future offers be linked in the filesystem?
- Should "Sell Architecture, Not AI" become a client conversation prep template?
- How should RitmaOS represent the removal test for business models?
- Should business flywheels connect to the same workspace architecture as technical build audits?
- What byproducts should RitmaOS automatically prompt users to capture after a client engagement?
