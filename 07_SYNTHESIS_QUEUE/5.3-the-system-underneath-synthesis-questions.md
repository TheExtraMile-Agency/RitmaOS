# Module 5.3 Synthesis Questions

Source module: Lesson 1.3 The System Underneath / Course Companion

## Questions

- Should training exercise outputs become a first-class RitmaOS data type?
- Should 60/30/10 Workflow Audit become a separate workflow from Applied Ratio Audit, or a variant of the same workflow?
- How should RitmaOS classify bottlenecks as missing infrastructure, missing orchestration, or missing AI capability?
- Should bottleneck diagnoses generate next-build opportunities automatically?
- Should leverage maps be calculators, markdown worksheets, or both?
- How should RitmaOS keep leverage estimates honest and avoid inflated automation claims?
- Should professional flywheels connect to partnership maps from Module 5.2?
- What filesystem structure best captures repeated bottlenecks and workflow patterns over time?
- Should tools-versus-patterns testing become part of onboarding?
- How should RitmaOS teach pattern ownership without becoming another static course?
- Should proof artifacts and next-client opportunities live inside the same workspace as workflow audits?
- How should repeated training data become reusable product specs without contaminating raw source material?

## Synthesis Direction

Module 5.3 should be synthesized with Modules 5.0, 5.1, and 5.2 because it reveals the repeatable diagnostic system underneath them: workflow audit, bottleneck layer diagnosis, training-as-intake, leverage mapping, flywheel design, and durable pattern learning.
