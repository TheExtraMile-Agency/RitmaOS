# Module 5.1 Synthesis Questions

Source module: Lesson 1.1 The Build — VigilOre / Course Companion

## Questions

- Should Applied Ratio Audit become a named RitmaOS workflow?
- Should Thin Wrapper vs Architectured System become a formal RitmaOS diagnostic?
- Should RitmaOS require every AI-system audit to start with a component breakdown?
- How should RitmaOS represent components that mix infrastructure, orchestration, and AI?
- Should the product push back when a user assigns more than 15% of a system to AI?
- Should AI necessity challenges become reusable prompt templates, product UI flows, or both?
- How should RitmaOS store pricing assumptions without treating them as final business decisions?
- Should pricing worksheets live beside architecture maps in the same workspace?
- How should RitmaOS distinguish legal advice from strategic contract-structure prompts?
- Should contract/IP strategy prompts be part of the core product or a service-layer add-on?
- What filesystem structure best reveals the 60% and 30% layers before AI is added?
- Should fragility test results be stored as a recurring health check for systems?
- How should RitmaOS connect model-provider fragility to workspace architecture and contract confidence?
- Should the product include a "move one level up" recommendation for thin-wrapper systems?

## Synthesis Direction

Module 5.1 should be synthesized with the earlier 60/30/10 material and Module 5.0 because it turns the applied ratio into executable audits: component breakdown, AI necessity challenge, value-gap pricing, contract/IP strategy, and model-provider fragility testing.
