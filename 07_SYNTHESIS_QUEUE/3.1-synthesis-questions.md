# Module 3.1 — Synthesis Questions

Source module: `02_RAW_MODULES/3.1-the-full-walkthrough-folder-architecture.md`
Status: Open questions for future synthesis.

## Architecture Questions

- Should RitmaOS standardize on the three-layer language from the video: Map → Rooms → Tools?
- Or should RitmaOS use the five-layer MWP model: global identity, workspace routing, stage contracts, reference material, working artifacts?
- Should the first RitmaOS template use rooms, stages, or both?
- What should be the minimum viable folder structure for a small business user?
- Should `CLAUDE.md` remain the primary root file, or should RitmaOS use a model-neutral name such as `AI.md`, `AGENT.md`, or `RITMA.md`?

## Product Questions

- Is the first product a downloadable folder template, a guided generator, a CLI, a web app, or a done-for-you service?
- What user type should get the first complete template: creator, freelancer, agency, developer, or small business owner?
- How much customization should happen during setup versus after the workspace is generated?
- Should RitmaOS include visual diagrams as part of the generated workspace documentation?
- Should the product include “before/after AI output” demos to prove the value?

## Workflow Questions

- What naming convention should RitmaOS enforce across all generated workspaces?
- Should RitmaOS create read/skip routing tables by default?
- Should each room have its own tool/skill map?
- Should every generated workspace include `/references`, `/output`, and `/drafts`, or should those vary by use case?
- Should the system include review gates as a default pattern even in non-stage-based workspaces?

## Business Questions

- Can “we build your AI folder system” be packaged as a clear agency service?
- What is the smallest paid deliverable that proves value within one client session?
- What parts of the method are generic enough for templates but still valuable enough to sell?
- Should RitmaOS emphasize “context engineering” or avoid that term for nontechnical customers?
